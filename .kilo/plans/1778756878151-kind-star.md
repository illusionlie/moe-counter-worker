# Theme Registry Refactor Plan

## Goal

Improve the existing theme call path so themes are easier to extend without scattering theme lookup, fallback, random selection, and metadata knowledge across `src/main.js`, `src/utils.js`, `themes/index.js`, and `src/index.html`.

The implementation should favor the smallest correct change first: add a stable theme registry API around the existing theme data, then optionally evolve the internal theme storage toward metadata/manifest-driven registration.

## Current Context

- Trellis current task: none.
- Active Trellis task present but unrelated: `00-bootstrap-guidelines/`.
- Git working tree: clean.
- Project mode: single-repo, frontend spec layer.
- Relevant guideline indexes read:
  - `.trellis/spec/guides/index.md`
  - `.trellis/spec/frontend/index.md`
- Build system: Webpack 5 with asset inline loaders for `gif` and `png`.
- Main code paths:
  - `themes/index.js` manually imports digit images and exports a raw object keyed by theme id.
  - `src/main.js` imports raw `themes`, performs `random` selection and fallback checks directly.
  - `src/utils.js` imports raw `themes`, performs its own fallback to `moebooru`.
  - `src/index.html` manually repeats the theme list in previews and in the tool select.

## Recommended Scope For First Implementation

Implement a low-risk registry layer without changing the image import strategy yet.

This gives immediate benefits:

- `main.js` no longer depends on raw theme object internals.
- `utils.js` no longer performs separate theme fallback logic.
- Theme resolution, fallback, random selection, and validation live in one module.
- Future manifest or metadata work can happen behind the same public API.

## Proposed Public Theme API

Create or expose these functions from `themes/index.js` or `themes/registry.js`:

```js
getTheme(themeId, fallbackThemeId)
hasTheme(themeId)
getThemeIds(options)
getRandomThemeId(options)
resolveThemeId(themeId, fallbackThemeId, options)
getThemeOptions(options)
```

Initial options can be minimal. If NSFW behavior is not intentionally changed in this pass, keep `random` behavior backward compatible by including all current themes.

Recommended internal constants:

```js
const DEFAULT_THEME = 'moebooru';
```

Recommended external behavior:

- `resolveThemeId('random', config.theme)` returns a valid random theme id.
- `resolveThemeId(undefined, config.theme)` returns a valid configured fallback or `DEFAULT_THEME`.
- `resolveThemeId('missing', config.theme)` returns a valid configured fallback or `DEFAULT_THEME`.
- `getTheme(id)` always returns a usable theme object.
- Invalid fallback config should fail safe to `DEFAULT_THEME`.

   - Capture this agreed MVP:
     - Add a registry API around existing theme data.
     - Update `main.js` and `utils.js` to consume the registry API.
     - Preserve current request compatibility.
     - Preserve current generated SVG behavior.
     - Do not require new theme asset conventions yet.
   - Note optional follow-ups:
     - Split themes into manifest plus per-theme image modules.
     - Add metadata such as `label`, `nsfw`, `hidden`, and `aliases`.
     - Add `/themes` API and dynamically render homepage theme options.

3. Start task after PRD is complete.
   - Command to run in execution mode: `python ./.trellis/scripts/task.py start <task-dir>`

4. Load development guidelines before writing code.
   - Use `trellis-before-dev`.
   - Relevant specs are currently sparse, but still load current frontend layer and thinking guides per workflow.

5. Add registry module.
   - Preferred minimal file: `themes/registry.js`.
   - Import existing raw theme object from `themes/index.js` or rename the raw export to avoid circular imports.
   - To avoid an import cycle, the cleaner minimal structure is:
     - Move current raw theme object to `themes/data.js`.
     - Make `themes/index.js` re-export registry functions and, if necessary for compatibility, a default raw theme object.
     - Make `themes/registry.js` import from `./data.js`.
   - Alternative smaller but less clean structure:
     - Keep `themes/index.js` as raw data default export.
     - Add `themes/registry.js` importing `./index.js`.
     - Update app code to import registry from `../themes/registry.js`.
   - Recommended choice: use `themes/data.js` plus `themes/registry.js` plus `themes/index.js` facade, because it creates a proper module boundary and prevents future cycles.

6. Update `src/main.js`.
   - Replace raw `themes` import with `resolveThemeId`.
   - Replace inline random and existence checks with:
     ```js
     theme = resolveThemeId(theme, config.theme);
     ```
   - Keep the rest of counter behavior unchanged.

7. Update `src/utils.js`.
   - Replace raw `themes` import with `getTheme`.
   - Replace:
     ```js
     const themeData = themes[theme] || themes.moebooru;
     ```
     with:
     ```js
     const themeData = getTheme(theme);
     ```
   - Keep rendering math and SVG output unchanged unless existing tests/build reveal an issue.

8. Add lightweight validation in registry.
   - Validate each registered theme has numeric `width`, numeric `height`, and exactly 10 images.
   - Prefer fail-fast at module initialization because missing digits produce broken SVG silently.
   - Keep validation simple and deterministic.

9. Consider homepage theme duplication as follow-up, not part of MVP unless explicitly requested.
   - Current `src/index.html` repeats theme options manually.
   - A future `/themes` endpoint can return `getThemeOptions()` and hydrate the select and previews.
   - Do not add client-side fetching in the first pass unless the user wants the UI list fixed now.

10. Verification.
   - Run `npm run build`.
   - If available and cheap, run a local Worker smoke check manually after build only if needed.
   - There is no configured lint/test/typecheck script in `package.json`; do not invent one.

11. Quality check.
   - Load `trellis-check` after implementation.
   - Verify spec compliance, build result, and that behavior remains compatible.
   - If issues are found, fix and re-run verification.

12. Finish workflow.
   - Use `trellis-update-spec` to decide whether the registry pattern or theme extension convention should be captured under `.trellis/spec/frontend/`.

## Backward Compatibility Requirements

- Existing URLs must keep working:
  - `/@demo?theme=asoul`
  - `/@demo?theme=moebooru`
  - `/@demo?theme=moebooru-h`
  - `/@demo?theme=rule34`
  - `/@demo?theme=gelbooru`
  - `/@demo?theme=gelbooru-h`
  - `/@demo?theme=random`
- Existing `config.yml` default theme behavior must keep working.
- Existing image dimensions and generated SVG output should remain effectively unchanged for valid themes.
- Invalid theme names should still fall back rather than erroring.

## Deferred Enhancements

These are good follow-ups but not required for the first registry refactor:

- Convert `themes/data.js` into a metadata-rich manifest.
- Add per-theme `index.js` files to localize the ten digit imports per theme directory.
- Add `label`, `nsfw`, `hidden`, `aliases`, and `sortOrder` fields.
- Add `/themes` API using `getThemeOptions()`.
- Update `src/index.html` to render theme select and preview list from `/themes`.
- Add `theme=random-safe` or `theme=random&nsfw=0` if random should exclude NSFW themes by default. This would be a behavior change, so it should be explicit.

## Risks And Mitigations

- Risk: accidental circular import between `themes/index.js` and registry.
  - Mitigation: use `themes/data.js` as raw data source, `themes/registry.js` as logic, `themes/index.js` as public facade.

- Risk: changing `random` semantics breaks existing users.
  - Mitigation: preserve current random behavior in MVP and document NSFW-safe random as follow-up.

- Risk: validation throws in production because existing theme data has a hidden issue.
  - Mitigation: validation should be tested by `npm run build`; if current data is invalid, fix data or reduce validation strictness only with a documented reason.

- Risk: static homepage list remains duplicated after registry refactor.
  - Mitigation: accept this as an explicit follow-up to keep MVP small and low-risk.

## Success Criteria

- `src/main.js` does not import raw `themes` or call `Object.keys(themes)`.
- `src/utils.js` does not import raw `themes` or duplicate fallback logic.
- Theme lookup and fallback are centralized in one registry module.
- Build passes with `npm run build`.
- Existing theme query parameters continue to work.
- The design leaves a clear path to manifest metadata and `/themes` API without rewriting callers again.
